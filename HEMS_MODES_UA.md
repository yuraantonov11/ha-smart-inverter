## 1. Огляд архітектури

Кожен тик HEMS (періодичний виклик із `AppStateProvider`) проходить пріоритетний ланцюжок:

```text
[Safety floor: SOC ≤ 22%?]
       ↓ НІ
[Manual override hold активний?]
       ↓ НІ
[Battery keepalive спрацював?]
       ↓ НІ
[Нічне вікно (23:00–07:00)?]
       ↓ НІ
[Realtime PV surplus ≥ 250W і SOC ≥ 30%?]
       ↓ НІ
[Forecast simulation → дефіцит?]
       ↓
[Рішення: SBU або USB]
```

### Пріоритети виходу (`outputSourcePriority`)

| Значення | Назва | Значення для системи |
|---|---|---|
| `'0'` | USB | Пріоритет мережі (Grid First) |
| `'2'` | SBU | Пріоритет сонця/АКБ (Solar/Battery First) |

### Пріоритети зарядки (`chargerSourcePriority`)

| Значення | Назва | Значення для системи |
|---|---|---|
| `'1'` | SNU | Заряд від сонця + мережі |
| `'2'` | OSO | Заряд тільки від сонця |

## 2. Спільні правила безпеки (завжди активні)

### 2.1 Жорсткий поріг SOC

**Тригер:** `batterySoc <= reserveSoc + 2` (за замовчуванням `<= 22%`)  
**Дія:**
- `output -> USB`
- `charger -> SNU`
- лог: `reason=safety_low_soc`

**Приклад:**
```text
SOC=21%, PV=1500W, 13:00
→ безпека має пріоритет над усім
→ USB + SNU
```

Причина: на низькому SOC не можна агресивно розряджати батарею, навіть якщо зараз є сонце.

## 3. Режим A — Adaptive (Smart)

Гібридний режим: realtime + прогноз.

- realtime шар: реагує на поточні `pvPower`, `loadPower`, `surplus`
- прогнозний шар: оцінює ризик дефіциту до 23:00

### 3.1 Ніч (23:00–07:00)

- `output -> USB` (стабільне правило)
- `charger` обирається прогнозом на завтра:
  - якщо дефіцит `> 0` -> `SNU`
  - якщо дефіцит `== 0` -> `OSO`

**Приклад:**
```text
02:00, SOC=65%, завтра хмарно -> дефіцит>0
→ USB + SNU (нічний дешевий тариф)
```

### 3.2 День (07:00–17:00)

Спочатку завжди `charger -> OSO`.

#### Крок 1: Realtime surplus (ключове покращення)

Обчислення:
```text
surplus = pvPower - loadPower
```

Якщо `pvPower > 80W`, `surplus >= 250W`, `SOC >= 30%` -> негайно `SBU`.

**Приклад:**
```text
13:00, PV=2500W, Load=900W, SOC=72%
surplus=1600W >= 250W
→ SBU (reason=pv_surplus_1600W_soc_72)
```

#### Крок 2: Forecast fallback

Якщо realtime не дав чіткого рішення, запускається симуляція до 23:00:
- `deficit == 0` -> `SBU`
- `deficit > 0` і `surplus <= 50W` і `SOC < 50%` -> `USB`
- інакше -> **утримати поточний режим** (без зайвого перемикання)

### 3.3 Вечір (17:00–23:00)

Вечір більш консервативний для захисту резерву:
- `charger -> OSO`
- якщо є ризик просідання (дефіцит/SOC/низький запас) -> `USB`
- якщо запас достатній -> `SBU`

**Приклад (захист резерву):**
```text
20:00, SOC=28%, сонця майже немає, дефіцит>0
→ USB (reason=evening_reserve_def_...)
```

## 4. Режим B — Night Arbitrage

Простий тарифний режим.

### Ніч (23:00–07:00)
- `output -> USB`
- `charger -> SNU`

### День (07:00–23:00)
- `charger -> OSO`
- `output -> SBU` лише коли є реальний surplus (`PV>80`, `surplus>=250`, `SOC>=30`)
- інакше режим не форсується (hold)

**Коли обирати:** якщо хочеш передбачувану поведінку без складної прогнозної логіки.

---

## 5. Режим C — Storm / Reserve

Режим максимальної готовності до блекаутів:
- `output -> USB` (`force=true`)
- `charger -> SNU` (`force=true`)

Мета: швидко набрати 100% SOC із PV + мережі.

**Коли вмикати:** перед штормом/відключенням.  
**Коли не треба:** у звичайні сонячні дні (може збільшити вартість енергії).

---

## 6. Наскрізно: Battery Keepalive

Проблема: деякі BMS "засинають", коли струм довго нульовий.

**Тригер:**
- активність батареї < 50W протягом 2 годин
- SOC > 22%
- режим не SBU

**Дія:**
1. Коротко `SBU` на 90 секунд
2. Повернення в `USB`
3. Скидання таймера неактивності

---

## 7. Наскрізно: Acoustic Comfort (нічна тиша)

- 22:00–07:00 -> `buzzerAlarmSetting=0`
- 07:00–22:00 -> `buzzerAlarmSetting=1`

Є dedup через `_lastAppliedBuzzer`, щоб не слати однакові команди щотик.

## 8. Anti-flapping і ручний override

### 8.1 Dwell (`minModeHold=20m`)

Після перемикання `USB<->SBU` нове протилежне перемикання блокується 20 хв, крім `force=true` (безпека/шторм).

### 8.2 Dedup (`commandDedupWindow=30s`)

Повторні однакові команди в межах 30с не надсилаються.

### 8.3 Manual override (`manualOverrideHold=30m`)

Якщо користувач змінив режим вручну (або натиснув кнопки в UI, які викликають `armManualOverride()`), алгоритм 30 хв не втручається в `output`.

## 9. Швидка таблиця параметрів `HemsTunables`

| Параметр | Default | Що робить |
|---|---:|---|
| `reserveSoc` | 20% | Нижній резерв батареї |
| `minOperatingSoc` | 30% | Нижче цього SBU менш бажаний |
| `midSoc` | 50% | Межа для рішень у дефіцитних сценаріях |
| `pvSurplusEnterW` | 250W | Поріг входу в SBU |
| `pvSurplusExitW` | 50W | Поріг виходу/умови USB при дефіциті |
| `minModeHold` | 20m | Захист від частих перемикань |
| `manualOverrideHold` | 30m | Скільки поважати ручний режим |
| `commandDedupWindow` | 30s | Придушення дубль-команд |

## 10. Схеми прийняття рішень

### Adaptive (день)

```text
SOC <= 22% ?
  YES -> USB + SNU (force)
  NO  -> manual hold active ?
            YES -> charger only / return
            NO  -> set charger OSO
                    -> realtime surplus strong ?
                        YES -> SBU
                        NO  -> forecast deficit to 23:00
                               deficit==0 -> SBU
                               deficit>0 && low surplus && low-mid SOC -> USB
                               else -> HOLD
```

### Adaptive (вечір)

```text
set charger OSO
simulate deficit to 23:00
if reserve risk -> USB
else if battery enough -> SBU
else HOLD
```

## 12. FAQ / Діагностика

### Чому при сонці буває USB?

Перевір `reason=` у логах.

| reason | Пояснення | Що робити |
|---|---|---|
| `safety_low_soc` | SOC дуже низький | Дочекатись відновлення SOC > 22% |
| `day_forecast_deficit_*_low_soc` | Алгоритм захищає вечірній резерв | Скоригувати `productionCoefficient`/пороги |
| `dwell active` | Спрацював anti-flap | Почекати завершення dwell |
| `manual hold active` | Активний ручний override | Дочекатися завершення hold |

### Чому після ручного SBU повертає назад?

Зазвичай через відсутній `manual hold` (коли зміна не зафіксована додатком) або завершення hold-вікна. У UI-кнопках `ControlPanel` це вже враховано через `armManualOverride()`.

## 13. v1.4 — Шар оптимізації HEMS v2 (НОВИНКА)

v1.4 розширює основний алгоритм 10 шарами оптимізації. **Зворотна сумісність гарантована** — всі нові можливості опціональні через `HemsOptimizationProfile`. За замовчуванням алгоритм ведеться як v1.3.

### 13.1 Динамічні часові вікна (астрономічні розрахунки)

Замість жорстких 07:00, 17:00, 23:00 система тепер вичисляє час сходу/заходу сонця на вашій локації.

```text
GPS: 50.45°N, 30.52°E (Київ)
Літо: День 05:30–20:30, вечір 20:30–23:00
Зима: День 08:00–16:30, вечір 16:30–23:00

Вплив: +5–10% ефективності за рік
```

### 13.2 Адаптивний поріг излишку PV

Поріг для входу в режим SBU тепер навчається з вариацій PV і змінюється на льоту.

| Умови | Поріг |
|---|---|
| Ясно (стійке вироблення) | 100–150W (низькій для швидшої реакції) |
| Хмарно (велика дисперсія) | 200–250W (високий для менше флепінгу) |

**Вплив:** -30% флепінгу на хмарні дні, краще використання сонячних днів.

### 13.3 Адаптивний таймер затримки (дWells)

Замість завжди 20 хвилин, затримка тепер мотається по історичним шаблонам:

- За день: ~10–12 хв на ясні дні (швидча реакція)
- На хмарні дні: 25–30 хв (менше коливань)

### 13.4 Здоров'я батареї і адаптивний резервний SOC

Резервний SOC тепер враховує вік батареї та деградацію:

- Молода батерея (< 2 років): резерв = 18% (агресивна)
- Стара батерея (5–10 років): резерв = 25% (захисна)

**Вплив:** +30–50% довголіття батареї, +10% корисної ємності на молодих системах.

### 13.5 Остаток на дешеві години (багатозонні тарифи)

Якщо у вас є час-залежні тарифи (3€ ніч, 8€ день), алгоритм тепер автоматично чекає дешевих годин.

```text
Ніч 23:00–07:00, тариф:
00:00–03:00 = 3€ (дешево)
03:00–07:00 = 5€ (середньо)

Якщо світанок приходить в 03:15 з тарифом 5€:
→ алгоритм може відсунути зарядку на 3 години раніше (00:00)
→ економія 2€/кВт·ч
```

**Вплив:** +15–25% скорочення тарифних витрат на багатозонні системи.

### 13.6 Прогнозування попиту на енергію

Замість плоскої оцінки (600 Вт·год/год завжди), система тепер пророкує споживання на основі:
- Часу доби
- Вихідних днів
- Сезону
- Історичних даних останніх 7 днів

**Вплив:** +10–15% точності прогнозу при переходах вихідні/робочі дні та сезонні перехода.

### 13.7 Свідомість до надійності мережі

Нові налаштування включають календар планових відключень мережі. Коли виявляється відключення у межах 6 годин:
- Алгоритм автоматично включає режим **Storm** (100% SOC передзарядка)
- Журнал: `reason=grid_outage_precharge`

### 13.8 Вибір стратегії (дропдаун у налаштуваннях)

Нова опція **Налаштування → Стратегія HEMS** для вибору пріоритету:

| Стратегія | Поведінка |
|---|---|
| **Економічна** | Мінімізація витрат на електроенергію |
| **Максимум сонця** | Максимум самоспоживання квітня |
| **Довголіття батареї** | Зменшення циклів, розширення ресурсу |
| **Надійність мережі** | Передзарядка перед планови відключеннями |
| **Гібридна** (за замовч.) | Баланс усіх чотирьох цілей |

### 13.9 Журналізація з кодами причин

Кожне рішення алгоритму містить машиночитаний код причини:

```
Приклад логу:
[14:30] 🌤️ HEMS: Surplus ≥ 250W → SBU (reason=surplus_enter_sbu, pv=1850W, load=450W, soc=78%)
[18:45] ⚠️ HEMS: Skip SBU, dwell active (reason=dwell_lock, elapsed=8min/20min)
[03:15] 💰 HEMS: Charging deferred to cheap window (reason=tariff_expensive_defer, next_cheap=00:00+2h)
```

Допомагає пошуком по ключовому слову діагностувати поведінку алгоритму.

### 13.10 Карточка діагностики параметрів

Нава вкладка **Налаштування → Діагностика** показує живі значення без режиму DEBUG:

## 14. Посібник міграції: v1.3 → v1.4

### 14.1 Зворотна сумісність (за замовчуванням)

**Nichts zu tun.** Якщо ви не налаштовуєте власний `HemsOptimizationProfile`, програма використовує білятки за замовч., які походять v1.3 поведінки:

- Часові вікна: фіксовані (07:00, 17:00, 23:00)
- Пороги PV: статичні (250W вхід, 50W вихід)
- Резерв SOC: 20%
- Затримка: завжди 20 хвилин
- Без тарифної економіки
- Без прогнозу попиту

**Результат:** Існуючі користувачи не помітять змін; алгоритм працює ідентично.

### 14.2 Включення функцій v1.4

Щоб користуватися v1.4, налаштуйте **Налаштування → Стратегія HEMS**:

**Крок 1: Виберіть стратегію**
- Відкрийте **Налаштування** → **Автоматизація**
- Виберіть **⚙️ Стратегія HEMS** з дропдауну
- Опції: Економічна, Максимум сонця, Довголіття, Надійність, Гібридна

**Крок 2 (Опец.): Локація**
- **Налаштування → Локація** — введіть координати
- (Автодетекція координат — планується на v1.5)

**Крок 3 (Опец.): TOU тариф**
- Якщо у вас багатозоний тариф:
  - **Налаштування → TOU ціни**
  - Визначте годинні ставки

**Крок 4: Моніторинг**
- Перевіряйте **Налаштування → Діагностика** для живих параметрів
- Шукайте у журналах keyword `reason=` для аналізу
